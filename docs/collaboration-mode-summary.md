# Magic Compass 协作模式重点摘要

## 核心结论

- 最优模式不是“完全自主”，而是“有边界的高自主施工”。
- GPT 负责判断与审稿，Codex 负责施工与交付，用户负责阶段拍板。
- 真正能降低返工的关键，不是更多人参与，而是：
  - 单一事实来源
  - 明确的范围闸门
  - 固定的回报格式

## 当前模式建议

### 适合先给 GPT 的事

- 需求整理与去歧义
- 视觉与文案判断
- 模块顺序确认
- 审稿与优先级排序
- 写给 Codex 的精确施工指令

### 适合直接给 Codex 的事

- 判断技术栈与现有实现
- 在 repo 内改文件
- build / preview / 截图
- 响应式微调
- 输出交付物
- 整理给 GPT 的审稿交接文件
- 同步 GitHub

### 必须由用户拍板的事

- 品牌方向变化
- 首页定位变化
- 模块顺序变化
- 合规、监管、风控表达
- 技术栈切换
- 范围扩大到新页面或新功能

## 未来模式建议

### Codex 可自主完成

- 环境安装与依赖管理
- 本地预览与浏览器截图
- 产出静态预览包
- 一键本地预览脚本
- 分支、提交、推送
- 按既定 brief 连续完成 1 到 2 轮施工

### Codex 应主动向 GPT 求助

- 缺文案
- 缺设计判断
- 需要审稿意见
- 需要图像生成提示词
- 需要结构优化建议

### 仍必须先找用户确认

- 品牌定位变化
- 合规风险
- 需要上线、登录、付费或连接外部正式资源
- 要大删内容或重构技术栈
- 外部建议与既有 brief 明显冲突

## 建议从现在开始建立的规则

- 每阶段一份 `TASK_BRIEF.txt`
- 每里程碑一份 `REVIEW_REQUEST.txt`
- 每轮固定三张图：
  - desktop
  - mobile viewport
  - mobile full-page
- 最终交付固定两种预览方式：
  - 可双击打开的静态预览包
  - 一键本地预览方案
- 运行说明优先使用 TXT
- 只记录里程碑决策，不记录所有微调噪音

## 一句话建议

当前阶段用“GPT 做判断、Codex 做施工、用户做拍板”的三层结构；未来升级成“有边界的高自主执行”，而不是“无限自主代理”。
