# Magic Compass 官網預熱專案定位說明

## 1. 專案定位

本專案目前定位為：

**官網改版預熱專案 / 工作流演練專案**

這一輪的主要目的，**不是一次性完成最終正式官網**，而是先把整套協作、審稿、開發、文檔沉澱與交接流程跑熟。

---

## 2. 為什麼要定義成「預熱專案」

目前屬於第一次正式以以下模式推進官網改版：

- 使用 ChatGPT 做需求整理、審稿、任務拆解
- 使用 Codex 進行前端施工、預覽、截圖、回報
- 使用 GitHub 存放討論摘要、審稿意見、協作交接文檔
- 以 `.md` 作為知識沉澱與後續可延續的專案記錄

在這個階段，如果直接將目標定義為「一次做出最終正式商用官網」，容易造成：

- 返工過多
- 對第一次協作模式抱有不切實際期待
- 因流程尚未成熟而浪費大量時間在溝通與修補
- 將本應該用來建立方法論的階段，誤判成最終交付階段

因此，本案更合理的定義是：

> 先把方法跑通，再把成品做深。

---

## 3. 本專案當前真正目標

### 核心目標
1. 建立「Ming → GPT → Codex」的穩定協作流程
2. 跑通 GitHub + `.md` 備份與交接機制
3. 驗證前端開發流程是否可行
4. 驗證 Vite / Vue / preview / screenshot / WAP 審稿流程
5. 累積可複用的提示詞、審稿格式、任務單格式
6. 產出一版具有參考價值的首頁原型
7. 為未來新主機正式專案建立可直接承接的知識庫

### 非核心目標
- 不強求這一輪一定完成最終正式官網
- 不強求一次到位
- 不強求所有頁面全部完成
- 不強求第一次就把人機協作模式磨到最順

---

## 4. 本專案的成功標準

本預熱專案若達成以下項目，即可視為成功：

### A. 工作流成功
- GPT 能穩定負責審稿、拆任務、整理文檔
- Codex 能穩定負責前端施工、預覽、截圖、回報
- Ming 能以監工角色控制節奏，而非陷入大量零碎傳話

### B. 文檔沉澱成功
- 每個大階段都有對應 `.md`
- 關鍵設計方向、修正意見、交接內容可追溯
- 未來可從這些 `.md` 直接做總整理與新專案承接

### C. 首頁原型成功
- 至少完成一版可審查的首頁前端原型
- PC / WAP 都有可比對版本
- 能明確知道目前與理想稿的差距在哪裡

### D. 問題被辨識成功
- 哪些地方是設計問題
- 哪些地方是素材不足
- 哪些地方是 Codex 實作問題
- 哪些地方是預覽 / 截圖 / 流程問題
- 哪些地方需要等未來新主機與更高權限後再發力

---

## 5. 本輪專案已驗證或正在驗證的重點

### 已有方向
- 首頁定位偏品牌形象與信任感
- 不推翻現有官網商業骨架
- 保留 CFD 平台感與市場感
- 不做重轉化 landing page
- 以結構保留型優化為主

### 已在演練的能力
- 視覺參考稿 → 前端首頁原型
- GPT 審稿 → Codex 修正
- WAP 與桌機雙端審查
- 將每輪結論沉澱為 `.md`
- 用 GitHub 當作專案中控台之一

### 目前也已發現的問題
- Private repo 導致 GPT 無法直接讀取 GitHub 內容
- Codex 手機端預覽 / 截圖流程仍不穩定
- 視覺稿、實作稿、審稿稿來回時，流程仍有些混亂
- 若無固定任務單格式，很容易在多輪協作中失焦

---

## 6. 為什麼要持續把大步驟整理成 `.md`

每一次大步驟整理成 `.md`，有幾個長期價值：

### 1. 降低記憶斷裂
每輪設計、審稿、修正、決策都能被保留下來，不用每次重講。

### 2. 降低返工
當 Codex 或未來新專案再回頭看時，能快速知道：
- 哪些已確認
- 哪些只是暫定
- 哪些被打回過
- 哪些是未來再優化項

### 3. 方便未來總整理
等未來新主機正式專案開啟後，可直接把所有 `.md` 集中整理：
- 視覺方向
- 工作流規則
- Codex 指令模式
- 首頁演進歷程
- 已知問題與解法
- 尚未完成但值得延續的內容

### 4. 讓預熱專案不會白做
即使本輪最終沒有完成正式官網，所有過程與結論也都會變成下一輪正式案的燃料。

---

## 7. 本專案與未來「新主機正式專案」的關係

本專案不是終點，而是：

## 未來正式專案的前置演練與知識累積器

未來若新主機到位、環境穩定、Codex 權限更完整，預計可另開新專案，作為：

**Magic Compass 官網正式專案**

到時可直接承接本預熱專案的成果，包括：

- 所有 `.md` 文檔
- 視覺方向判定
- 首頁模組順序
- 審稿標準
- 常用任務指令
- Codex 修正節奏
- 已知坑點與解法

這樣就能做到：

> 預熱專案負責摸路  
> 正式專案負責高速推進

---

## 8. 未來正式專案開啟時的建議做法

當新主機到位並正式開案時，建議流程如下：

### 第一步：集中讀取預熱專案所有 `.md`
由 GPT 讀取並整理所有歷史文檔，輸出總結版：
- 專案背景
- 設計方向
- 已確認原則
- 已知問題
- 已完成成果
- 待延續事項

### 第二步：建立正式專案規範
包括：
- Codex 長期協作規則
- 任務單模板
- 回報模板
- 截圖與預覽流程
- 素材與占位資產規則

### 第三步：正式推進多頁與高權限協作
等權限完整後，再導入：
- 自主取材
- 自主瀏覽器操作
- 自主圖片與資產補齊
- 更完整的頁面實作與測試

---

## 9. 本專案目前建議的工作原則

### 原則 1
先跑通流程，再追求完美

### 原則 2
有價值的結論才沉澱成 `.md`

### 原則 3
每輪只解一批問題，不同時開太多戰線

### 原則 4
首頁不是一定要本輪定稿，但工作流一定要跑熟

### 原則 5
預熱專案的價值在於方法論與知識沉澱，而不只是頁面成品

---

## 10. 接下來的執行方式

從本份文檔之後，建議延續以下模式：

### 每一個大步驟
由 GPT 協助整理為 `.md`

### `.md` 建議內容
- 本輪狀態
- 核心結論
- 保留項
- 必改項
- 可延後項
- 給 Codex 的施工指令
- 下一步建議

### 未來總整理時
再把全部 `.md` 交由 GPT 做一次全局總結，作為新主機正式專案的啟動資料。

---

## 11. 一句話總結

本專案的主要任務，不是保證第一次就把官網做到終局版，而是：

**把「GPT 審稿 + Codex 施工 + GitHub 文檔沉澱」這套流程先跑熟，並把所有可用經驗沉澱下來，為未來新主機正式專案鋪路。**
